# Band Sync & Jam Companion — Build Brief

A portable spec for prompting Claude Code. Hand it the whole thing, or paste sections as you build milestone by milestone.

---

## 1. Core vision

A shared, real-time web companion for a small acoustic ensemble that keeps everyone aligned on the *same tune at the same moment* — whether that tune is a fully notated arrangement or an open jam. Two modes, one shared state object, no fixed leader.

- **Notated mode** — a locked, synced playhead across all devices. Everyone sees their own instrument's part, scrolled/zoomed however they like, while the musical position stays in lockstep. Any member can grab tempo or transport (play/pause/seek) for the whole group.
- **Jam mode** — no playhead. A shared key/tempo/section/turn state keeps everyone conceptually aligned, backed by per-instrument reference panels (chords, scales, licks) and a compact song card (feel, form, solo order, reference track).

The guiding principle: **sync the musical position, not the screen.** Shared state is tiny and logical; everything visual is local.

---

## 2. Context

**Who it's for:** a 3–5 piece acoustic/old-time/bluegrass group.

- Primary user plays guitar, high-G *re-entrant* ukulele (tuning G4 C4 E4 A4 — strings will not be changed; capo is available), UBass, and occasionally electric cello (C–G–D–A, perfect fifths).
- A bandmate plays fiddle and reads standard notation.

**Existing assets to build on, not rebuild:**

- **Soundslice** (paid) with arrangements already authored — synced multi-part notation, per-part mute/solo, tempo, looping, shareable secret links.
- A **tune-arranger toolkit** (Python) that converts ABC melody + chords into 4-part **MusicXML** (fiddle in standard notation; guitar, uke, bass in tab) plus a chartbook PDF. Includes a **Viterbi/DP fingering optimizer** (campanella-aware), a string-map legend generator, and a LilyPond/music21 engraving pipeline. Instrument tunings and the high-G uke conventions are already encoded.
- A working **8-tune set** spanning reels, jigs, songs, and one chords-only copyrighted tune.

**The reusable engine that matters most for jam mode:** the fingering optimizer already solves "find playable notes within a hand-span on a given tuning, minimizing stretch." Pointed at *chord tones* instead of a melody, it generates chord voicings; pointed at *scale degrees*, it generates scale maps — for any tuning already defined (guitar, high-G uke, UBass, cello).

---

## 3. Unmet needs (what doesn't exist today)

1. **No real-time cross-device sync.** Soundslice playback is per-person; its practice lists are personal and account-bound. Nothing locks a band-wide playhead.
2. **No shared, multi-writer tempo** that *anyone* can adjust for *everyone*.
3. **No instrument-specific jam reference** — fretboard/fingerboard chord shapes, scale maps, and licks per key and per tuning — anywhere in the current toolchain.
4. **No unified shell** that holds a set list, switches between notated and jam modes, and carries one shared song-state.
5. **No shared "section pointer"** for improv ("we're on the B now / your solo next").

---

## 4. Feature set

### Shared state (the only thing synced)

- **Notated transport:** `{ playing, startTimestamp, startBar, tempo }` — current position is *computed locally* on every peer as `bar = startBar + tempo × (now − startTimestamp)`. Multi-writer: any peer can stamp new values.
- **Jam state:** `{ key, tempo, section, whoseTurn }` — also multi-writer; latency-tolerant (no one cares about 100 ms here).
- **Set list:** ordered list of tunes, each tagged with its mode (notated | jam) and its source (slice ID / MusicXML file).

### Local-only (never synced)

- Which instrument part is displayed.
- Scroll, pan, zoom.

### Per-instrument reference panels (jam mode)

- **Chord diagrams** — generated by reusing the fingering optimizer over chord tones, per tuning. (Note: uke chord *shapes* are identical for re-entrant vs. linear tuning — the high G only changes which octave sounds, not the fingering. The re-entrant quirk only matters for scales/licks.)
- **Scale maps** — scale degrees of the current key rendered on each instrument's fretboard/fingerboard as SVG.
- **Lick library** — a small *curated* set per style and key (this part is human-authored vocabulary, not computable), rendered through the existing MusicXML/LilyPond pipeline. Scope is bounded: old-time/bluegrass live on major, mixolydian, and pentatonic/blues — roughly five scale choices and I–IV–V(–vi) per key.

### Song card (jam mode)

Key, tempo, a one-line feel descriptor ("mixolydian shuffle, leave space, trade after two"), form/roadmap (head–solos–out), solo order, and a reference-track link. Pairs with the shared section pointer so the whole card advances together when someone calls the turnaround.

## 5. Sync layer & "host migration"

**Host migration is designed away, not solved.** Because the transport is a small deterministic state object and every peer computes position independently, *no one is the clock during playback*. A peer is only needed at the instant of a change (to stamp new values), which is just a write. Make those writes last-write-wins by timestamp and the peers are symmetric — nothing to migrate, late joiners pull current state from any neighbor, a peer dropping is a non-event.

**Recommended P2P stack:** **Yjs (CRDT) + y-webrtc**, or **Trystero** (brokers peer introductions over public infrastructure so you don't run signaling yourself). You get conflict-free merge, automatic late-joiner state transfer, and presence essentially for free.

**Honest caveats:**

- WebRTC needs a rendezvous to cold-start (signaling). Lightweight, but it exists — use a public/trackerless option to stay maintenance-free.
- NAT traversal: STUN covers the common case; symmetric NATs need a **TURN relay**, at which point "P2P" quietly becomes "relayed." Same-room/same-WiFi rehearsal is the easy case (direct LAN, fully offline-capable). Remote jamming is where TURN starts to matter.
- **Clock sync is the real hard dependency, not host election.** For jam state it's a non-issue. For the tight notated playhead, add a small NTP-style handshake (peers ping, estimate round-trip + offset, carry a shared clock) and project forward on receipt ("bar 17 *as of* timestamp T" → where it's advanced to by arrival).

**Skip P2P only for pure same-room rehearsal:** a ~30-line WebSocket relay on someone's laptop over the LAN is even less code and offline by default. P2P earns its keep mainly for different-house jamming.

## 6. Build approaches (pick one to start; #3 is the likely end state)

**A. Soundslice-embed shell + realtime channel — fastest to working.**
Reuses every arrangement you already own. Embed with `api=1`; listen for player events (`ssPlay` carries current time, plus `ssPause`, `ssSeek`, `ssCurrentBar`, `ssSpeed`); drive followers with `seek`, `setSpeed`, `play`/`pause`; poll `getCurrentTime` periodically to drift-correct. Followers run a silent slice so the playhead moves without audio phasing. **Tradeoff:** Soundslice owns its audio clock, so you soft-seek to correct drift — tight enough for follow-along, not sample-accurate. Sync by **bar number**, not pixel scroll, so per-instrument slices line up.

**B. alphaTab custom renderer + logical-clock sync — tightest lock, most control.**
Renders your existing MusicXML (notation + tab, per-track solo/mute, local zoom). You own the clock, so the cursor is a pure function of the shared clock — no drift, instant tempo snaps — and there's no per-bandmate Soundslice dependency. **Tradeoff:** more to build (player wrapper + clock + sync).

**C. Hybrid — realistic end state.**
Soundslice (or alphaTab) embeds for notated tunes; custom panels for jam mode; one shared song-state shell unifying both. The shell isn't extra work — it's the thing that was always going to tie the set together, and jam mode *requires* custom UI (fretboards, scale maps, licks) regardless of renderer choice.

## 7. Suggested milestone sequence for Claude Code

Build in this order so each step is demoable and de-risks the next:

1. **Shared-state core (no audio).** Two browser tabs, leaderless CRDT over Trystero/y-webrtc, a tempo number and a section pointer that either tab can change and both reflect. Proves the sync model.
2. **Clock-sync handshake.** Add the NTP-style offset exchange; display each peer's agreed "now." Proves tight timing is achievable before any renderer is involved.
3. **Notated mode, one renderer.** Wire in Soundslice embeds (path A) or alphaTab (path B). Slave the cursor to the computed position; add multi-writer transport. Test the drift correction with 2–3 devices.
4. **Set list + mode switch.** Ordered shared set list; tapping a tune loads its renderer or its jam card.
5. **Jam panels — one instrument first.** Generate chord diagrams + scale map for, e.g., uke or UBass in G via the fingering optimizer. Render as SVG. Then generalize across tunings.
6. **Song card + lick library.** Author a handful of licks per key/style; render through the MusicXML/LilyPond pipeline; attach feel/form/solo-order metadata.

Stub aggressively: hardcode one tune and one instrument until the loop works, then widen.

## 8. How to prompt Claude Code

- **Pin the stack up front** so it doesn't wander: "Use Yjs + Trystero for leaderless P2P state. No host, no leader election — state is last-write-wins CRDT. Sync musical position by bar, computed locally from `{startBar, startTimestamp, tempo}`."
- **Give it the real inputs.** Hand it 2–3 sample MusicXML files from the tune-arranger toolkit, the instrument tuning definitions, and the fingering-optimizer module — so chord/scale generation reuses your engine instead of reinventing it.
- **Be explicit about the local/synced split** every time UI comes up: "Scroll, zoom, and instrument selection are local per device and must never be written to shared state."
- **Name the renderer decision explicitly** (path A vs B) — they have different APIs and Claude Code should commit to one per build.
- **Carry the gotchas** (Section 9) into the prompt so they're handled, not rediscovered.

## 9. Key technical facts & gotchas

- **Soundslice JS API:** enable with `api=1` on the *full* embed (not miniplayer). Methods include `seek` (seconds), `setSpeed` (float, 1 = normal), `play`/`pause`, `getCurrentTime`. Events include `ssPlay` (carries current time), `ssPause`, `ssSeek`, `ssCurrentBar`, `ssSpeed`. A slice with no recording still moves the playhead silently — ideal for followers.
- **High-G re-entrant uke:** string 4 is the *bottom* tab line but the *highest* sounding open string (G4). Chord shapes are unaffected; scales and licks must respect the actual sounding octave. The string-map already encodes this — reuse it.
- **Cello in fifths (C–G–D–A):** adjacent open strings already form root–fifth pairs, so most set keys (C, G, D) need little or no fingering for bass-style jam work.
- **Copyright:** at least one tune in the set is chords-only (no melody notation) by design. Keep that flag in the data model so the app never renders a melody for it.
- **Latency projection** is what makes "locked" actually locked: always interpret a transport message as a value *as of its timestamp*, then advance it by the elapsed time on arrival.
- **Files don't persist between Claude sessions** — re-supply source files (MusicXML, optimizer, tuning defs) when continuing work.
